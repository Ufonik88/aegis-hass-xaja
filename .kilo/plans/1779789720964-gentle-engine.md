# Video Functionality Analysis & Implementation Plan for Aegis for Ajax

## Project Overview

**Aegis for Ajax** is a third-party Home Assistant custom integration for Ajax Security Systems. It communicates with Ajax cloud servers via three channels:

| Channel | Protocol | Purpose |
|---------|----------|---------|
| gRPC | TLS to `mobile-gw.prod.ajax.systems:443` | Device state, arm/disarm, photo capture |
| HTS | Proprietary binary over TCP+TLS | Hub network metrics, electrical readings |
| FCM Push | Firebase Cloud Messaging | Security events, photo URLs, doorbell rings |

The integration uses **reverse-engineered protobuf definitions** (extracted from the Ajax Android APK) to communicate via the same gRPC API as the official Ajax mobile app.

---

## Current State: Photo-Only Camera Support

The integration currently supports **only still-image capture (Photo on Demand)** for a limited subset of camera devices:

```python
CAMERA_DEVICE_TYPES = {"motion_cam", "motion_cam_outdoor", "motion_cam_fibra",
                       "motion_cam_phod", "motion_cam_outdoor_phod", "motion_cam_fibra_base"}
```

The photo data flow:

```
Button press → gRPC capturePhoto → wait for FCM push → extract photo URL from FCM →
gRPC streamNotificationMedia → download via HTTP → save to disk → display in HA Camera entity
```

## What Is Missing: Video Streaming

**Zero video streaming exists.** Specifically:

1. **`AjaxCamera` (`camera.py`)** only overrides `async_camera_image()` (returns still bytes). It does NOT override `stream_source` / `async_stream_source()` — required by HA to ingest RTSP/HLS streams.

2. **`video_edge_*` devices are invisible** to the camera platform. Devices like Ajax Doorbell, Indoor Cam, Turret, Bullet, MiniDome, and third-party ONVIF cameras bridged through VideoEdge NVR are parsed by `DevicesApi` but never create `camera` entities — they only appear as binary sensors.

3. **No video API calls are made.** Despite having fully compiled protobuf stubs for:
   - `WebrtcService.initiate` (WebRTC live streaming)
   - `StreamVideoPlayerService.execute` (gRPC video player streaming)
   - `GetVideoEdgeOnvifAndRtspSettingsService.execute` (ONVIF/RTSP config)
   - `CloudArchiveService.streamVideoFragmentsData` (archive playback)
   
   …none of these are called anywhere in the integration.

4. **`MediaApi` (`api/media.py`)** is photo-only — a single method `get_photo_url()`. No video stream URL retrieval.

5. **No `Stream` component integration.** HA's built-in `Stream` component can transcode RTSP/HLS to HLS for browser playback. The integration never provides it a stream URL.

## What the Proto Definitions Reveal About Ajax's Video Architecture

The reverse-engineered protos (~70+ video-related files) describe two distinct video subsystems:

### A. Modern VideoEdge System (Ajax NVR + native cameras)

Uses **WebRTC** as the primary streaming protocol through Ajax's cloud relay infrastructure:

```
[ONVIF/RTSP camera] → [Ajax VideoEdge NVR] → [Ajax Cloud (TURN/STUN)] → [Client WebRTC]
```

Key services available:
- `WebrtcService.initiate` — server-streaming, returns ICE servers, SDP offers/answers
- `StreamVideoPlayerService.execute` — unary-stream, gRPC-based video delivery
- `CloudArchiveService.streamVideoFragmentsData` — pre-signed MP4 URLs or protobuf fragments
- `VideoEdgeService.streamUpdates` — real-time device updates
- `GetVideoEdgeOnvifAndRtspSettingsService.execute` — RTSP/ONVIF port configuration

**Critical finding: Ajax does NOT expose raw RTSP/HLS/MJPEG URLs for VideoEdge devices.** All live video flows through WebRTC via Ajax's cloud. The `rtsp_http_port` on VideoEdge appears to be an internal/external RTSP server configuration, not a direct URL for client streaming.

### B. Legacy Hub Surveillance Camera System

For older hub-connected cameras, a separate `SurveillanceCameraStreamSettings` proto provides:
- `stream_data_url` — a raw RTSP URL string (for `RTSP_STREAM` service type)
- Brand-specific OAuth credentials (Hikvision, Safire, Dahua, Uniview)

This is the **only place in the entire API where a raw RTSP URL is returned directly** to the client.

## Device Types Currently Parsed but Not Used for Video

The `DevicesApi` parser recognizes these video device types via the `video_edge_channel` oneof:

| Internal Type | Description |
|---------------|-------------|
| `video_edge_doorbell` | Ajax DoorBell (also appears as `motion_cam_video_doorbell`) |
| `video_edge_turret` | TurretCam |
| `video_edge_bullet` | BulletCam |
| `video_edge_minidome` | MiniDome |
| `video_edge_indoor` | Indoor Cam |
| `video_edge_unknown` | Unknown video edge type |

Plus third-party ONVIF cameras bridged through `MediaDevice` entries on the NVR.

Status fields like `camera_view`, `someone_has_camera_stream_access`, and `someone_can_make_photo` are present in the proto but **never parsed**.

## Implementation Approaches

### Approach A: WebRTC-based (matches Ajax's native method)

**Pros:** Works with Ajax's cloud architecture, supports live + archive, supports dual-stream (main/sub), works for both native and third-party cameras.

**Cons:** 
- Requires full WebRTC signaling implementation (ICE, SDP exchange)
- Requires WebRTC client support in HA (HA has limited WebRTC camera support — mostly through external integrations like go2rtc/WebRTC Camera)
- Additional dependency on a WebRTC stack
- Most technically complex

### Approach B: RTSP passthrough via ONVIF/RTSP settings

**Pros:** Direct RTSP stream, works natively with HA's Stream component, low latency, no cloud dependency.

**Cons:**
- Only works for VideoEdge devices that have RTSP enabled (may require configuration via a secondary call)
- Requires `get_video_edge_onvif_and_rtsp_settings` to return usable RTSP URLs
- May not work for all camera types
- Requires ONVIF credentials management

### Approach C: gRPC StreamVideoPlayer (video player streaming)

Leverage `StreamVideoPlayerService.execute` to get video frames via gRPC server-stream.

**Pros:** Uses existing gRPC infrastructure, no new protocol dependencies.

**Cons:**
- Response format unclear — is it raw frames? Encoded video chunks? WebRTC offers?
- May not integrate easily with HA's Camera/Stream architecture
- Unknown performance/encoding characteristics

### Recommended Hybrid Approach

1. **Primary: RTSP/ONVIF path** — Query `get_video_edge_onvif_and_rtsp_settings` to obtain RTSP URLs/credentials for VideoEdge channels. Use HA's built-in `Stream` component for transcoding.

2. **Secondary: Legacy hub camera RTSP** — Query `SurveillanceCameraStreamSettings` to get `stream_data_url` for hub-connected cameras.

3. **Fallback: Still-image previews** — Use `Channel.channel_preview.image_url` for devices where live streaming isn't available.

4. **Future: WebRTC** — Once the integration has a stable streaming platform, add WebRTC support for native Ajax camera streaming through the cloud.

### Recommended Phased Implementation

**Phase 1 (MVP): RTSP-based streaming for VideoEdge + hub cameras**
- Retrieve RTSP URLs via `get_video_edge_onvif_and_rtsp_settings` and `SurveillanceCameraStreamSettings`
- Create camera entities for `video_edge_*` device types
- Override `stream_source` on `AjaxCamera`
- Expose snapshot/preview URLs as fallback
- **Effort: ~10-15 days**
- **Risk: Medium** — depends on RTSP URL availability from Ajax API

**Phase 2: WebRTC live streaming**
- Implement `WebrtcService.initiate` signaling
- Integrate with HA's WebRTC camera support
- Support dual-stream (main/sub)
- **Effort: ~10-15 days**
- **Risk: High** — dependent on WebRTC API documentation

**Phase 3: Cloud archive + advanced features**
- Implement archive playback
- Add PTZ control if available
- Two-way audio
- Detection zone configuration
- **Effort: ~10-20 days**
- **Risk: Medium**
